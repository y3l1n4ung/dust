# HttpClient Specification: Parsing & Isolate Strategy

This document defines how Dust handles JSON serialization and off-thread execution.

## 1. Parsing Strategy Overview

Dust supports two modes of execution for response decoding, configured via `DustParseThread`:

1.  **`main`**: Decoding happens on the thread that initiated the request. Suitable for small payloads or non-Flutter CLI apps.
2.  **`isolate`**: Decoding is offloaded to a background isolate. **Required** for large payloads in Flutter to maintain responsiveness.

## 2. Modern Isolate Execution (`Isolate.run`)

For `DustParseThread.isolate`, Dust uses the modern `Isolate.run()` API.

### Why `Isolate.run`?
- **Simplicity**: Handles port setup and cleanup automatically.
- **Performance**: Direct communication with the background isolate without the overhead of manual `ReceivePort`.

## 3. Generator Logic

For every model `T` returned by an API, the generator performs the following (provided the model adheres to the **Model Contract** defined in the Overview):

### A. Main Thread Decoding
Decoding happens directly in the generated method by calling `T.fromJson(response.data!)`.
...
### B. Isolate Decoding
To use `Isolate.run`, the decoding logic must be a static or top-level function. The generator will emit a private helper:

```dart
// Generated at top-level of .g.dart
T _$decodeT(dynamic json) => T.fromJson(json as Map<String, dynamic>);

// Generated inside method
final decoded = await Isolate.run(() => _$decodeT(response.data!));
```

## 4. Collection Handling

If the return type is `List<T>`, the helper function handles the mapping:

```dart
List<T> _$decodeTList(dynamic json) {
  return (json as List).map((i) => T.fromJson(i as Map<String, dynamic>)).toList();
}

// Inside method
final decoded = await Isolate.run(() => _$decodeTList(response.data!));
```

## 5. Generic Models & Nullability

- **Generics**: For generic models like `Future<Paginated<User>>`, the generator emits specific decoders (e.g., `_$decodePaginatedUser`) to ensure type safety within the isolate boundaries.
- **Nullability**: 
  - If the return type is `Future<T?>`, the generated code handles `response.data == null` by returning `null`.
  - If a model field is nullable, the `T.fromJson` implementation (likely generated by Dust Serde) handles it.
- **Primitive Responses**: For primitive returns like `Future<String>` or `Future<int>`, decoding happens on the main thread to avoid isolate overhead, regardless of the `parseThread` setting.

## 6. Type Safety & Validation

- **Isolate Safety**: All data passed to `Isolate.run` must be sendable. Since `response.data` from Dio is already a primitive-based Map/List, it is safe.
- **Method-Level Overrides**: A request can be marked for isolate parsing via `@HttpParse(thread: DustParseThread.isolate)`, even if the class default is `main`.
