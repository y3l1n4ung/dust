# i18n

Dust i18n starts with generated Flutter bootstrap, static key scanning, ARB
reconciliation, and a small runtime API in `dust_flutter`.

---

## Installation

```yaml
dependencies:
  dust_flutter: ^0.1.0
  flutter_localizations:
    sdk: flutter
```

---

## Key Format

Use ARB-safe identifiers in Dart and ARB files. Keys must start with a letter
and contain only letters, numbers, or underscores.

| Dart key | Namespace | ARB file | ARB message key |
| :--- | :--- | :--- | :--- |
| `shop_title` | `shop` | `assets/i18n/en/shop.arb` | `title` |
| `shop_product_price` | `shop` | `assets/i18n/en/shop.arb` | `product_price` |
| `login_submit_button` | `login` | `assets/i18n/en/login.arb` | `submit_button` |

The namespace comes from loaded ARB bundle names. Dust uses the longest
matching loaded namespace prefix, so `shop_product_price` maps to namespace
`shop_product` only when `shop_product.arb` is loaded.

## Generated Bootstrap

Configure locales in `dust.yaml`. The first locale is the fallback locale.

```yaml
i18n:
  locales: [en, my]
```

Run `dust build` to generate:

```txt
lib/i18n/app_i18n.g.dart
```

Wrap the app with generated `AppI18n`:

```dart
import 'i18n/app_i18n.g.dart';

void main() {
  runApp(
    const AppI18n(
      child: App(),
    ),
  );
}
```

`AppI18n` creates `I18nController`, loads ARB assets from Flutter's asset
manifest, provides `I18nScope`, and exports Flutter-ready locale constants.

## Static Scan

Run the scanner to list statically discoverable keys:

```sh
dust i18n scan
```

`dust i18n scan` reports unique static keys, namespaces, default text, and
placeholder names. It ignores `TranslatedText.dynamic(...)` because those keys
come from API or model data at runtime.

## ARB Build

Run the i18n build command to scan Dart, update ARB files, and refresh the
generated bootstrap:

```sh
dust i18n build
```

The fallback locale is the first configured locale. Missing fallback messages
use `defaultText` when present, otherwise the full key. Missing non-fallback
messages are written as empty strings.

Dust preserves existing translations and metadata. Missing metadata is
backfilled with `description` and placeholder `example` fields. Dust does not
guess placeholder `type`; add richer ARB metadata manually when needed.

Run the no-write checker in CI:

```sh
dust i18n check
```

`dust i18n check` validates ARB files against the static scan. It fails on
invalid JSON, missing files, missing keys, missing metadata, empty
translations, placeholder mismatches, and missing Flutter asset declarations.
It warns on same-as-fallback translations and stale-looking keys. `dust i18n
build` reports missing asset declarations as warnings so first-time ARB
generation is not blocked.

## Runtime Setup

```dart
import 'package:dust_flutter/i18n.dart';
```

Assets use this default layout:

```txt
assets/i18n/{locale}/{namespace}.arb
```

With the default layout, generated `AppI18n` calls `loadAssetBundles()` and
discovers namespaces from Flutter's asset manifest.

Declare the files or locale directories in `pubspec.yaml`:

```yaml
flutter:
  assets:
    - assets/i18n/en/
    - assets/i18n/my/
```

Flutter only includes files directly inside a declared directory. With Dust's
default nested locale layout, `assets/i18n/` is not enough. Use each locale
directory or list each ARB file explicitly.

## Flutter App Shell

Inside the app, read `I18nScope` to wire Flutter's own localization fields from
the same controller. Use the generated constants instead of hand-writing
delegates and locale lists.

```dart
class App extends StatelessWidget {
  const App({super.key});

  @override
  Widget build(BuildContext context) {
    final controller = I18nScope.of(context);
    return MaterialApp.router(
      locale: appI18nLocaleOf(controller.locale),
      supportedLocales: appI18nSupportedLocales,
      localizationsDelegates: appI18nLocalizationsDelegates,
      routerConfig: router.config,
    );
  }
}
```

`TranslatedText` handles Dust messages. The `MaterialApp` fields handle
Flutter's built-in localized widgets, text direction, and locale-aware
framework behavior. Generated locale values use `Locale.fromSubtags`, so
language, script, and country codes such as `en_US` and `zh_Hans_CN` are
represented correctly.

## ARB Files

`assets/i18n/en/shop.arb`

```json
{
  "@@locale": "en",
  "@@context": "Translations for `shop` namespace.",
  "title": "Shop",
  "@title": {
    "description": "Translation for `shop_title`."
  },
  "product_price": "${price}",
  "@product_price": {
    "description": "Translation for `shop_product_price`.",
    "placeholders": {
      "price": {
        "example": "9.99"
      }
    }
  }
}
```

`assets/i18n/my/shop.arb`

```json
{
  "@@locale": "my",
  "@@context": "Translations for `shop` namespace.",
  "title": "ဆိုင်",
  "@title": {
    "description": "Translation for `shop_title`."
  },
  "product_price": "US$ {price}",
  "@product_price": {
    "description": "Translation for `shop_product_price`.",
    "placeholders": {
      "price": {
        "example": "9.99"
      }
    }
  }
}
```

ARB metadata keys such as `@@locale` and `@product_price` are ignored at
runtime.

## UI Usage

```dart
const TranslatedText(
  'shop_title',
  defaultText: 'Shop',
)
```

```dart
TranslatedText(
  'shop_product_price',
  defaultText: r'${price}',
  args: {'price': price.toStringAsFixed(2)},
)
```

Use `context.tr(...)` when a `String` is needed instead of a widget:

```dart
TextField(
  decoration: InputDecoration(
    hintText: context.tr(
      'shop_search_hint',
      defaultText: 'Search products',
    ),
  ),
)
```

Use `TranslatedText.dynamic(...)` when API, CMS, or remote config data contains
a translation key. Static scanning ignores dynamic keys.

```dart
TranslatedText.dynamic(
  promo.titleKey,
  fallback: promo.fallbackTitle,
)
```

Do not invent product keys for plain catalog APIs. Product title and
description should come from the backend in the requested locale, or render as
raw backend content when the API is not localized.

## Remote Overrides

Fetch remote ARB or JSON with your existing HTTP client, then apply the response
body as runtime overrides. Dust parses string messages, ignores ARB metadata,
and rebuilds widgets that read `I18nScope`.

```dart
final body = await api.getText('/i18n/en/shop.arb');
I18nScope.of(context).applyOverrideBundle(
  body,
  locale: 'en',
  namespace: 'shop',
);
```

The remote payload uses namespace-local ARB keys:

```json
{
  "title": "Remote Shop",
  "@title": {
    "description": "Ignored at runtime"
  }
}
```

---

## Lookup Order

Dust resolves a key in this order:

1. Runtime override.
2. Active locale bundle.
3. Fallback locale bundle.
4. `defaultText` or dynamic `fallback`.
5. The key itself.

Placeholders use `{name}` syntax. v0.1 does not implement ICU plural or select
messages yet.
